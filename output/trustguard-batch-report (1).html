Executive Summary

PyJWT (v2.6.0) is a widely used Python library for JSON Web Token (JWT) handling, but it has unresolved HIGH-severity vulnerabilities. Avoid using versions below 2.12.0 due to critical RFC non-compliance and key confusion risks. Upgrade immediately if JWT security is a priority.

Secure Code Review

No source code was provided for direct review. However, the vulnerability data reveals critical implementation flaws in PyJWT's handling of JWT standards. Specifically, the library fails to validate the `crit` (Critical) Header Parameter (RFC 7515 §4.1.11), allowing attackers to bypass security controls by including unrecognized extensions. Additionally, key confusion vulnerabilities (e.g., GHSA-ffqj-6fqr-9h24) enable attackers to exploit symmetric/asymmetric key mismatches, leading to forged tokens. These issues are confirmed in the provided vulnerability reports and PoCs.

🔍 Security Findings 1 CRITICAL · 2 HIGH · 1 LOW
1. Unpatched `crit` header bypass
CRITICAL
DEPENDENCY_CVE ✓ Confirmed
PyJWT fails to validate the `crit` (Critical) Header Parameter as required by RFC 7515 §4.1.11. This allows attackers to include unrecognized extensions in JWTs, bypassing security policies (e.g., MFA, token binding). The vulnerability is confirmed in versions up to 2.11.0 and fixed in 2.12.0.
GHSA-752w-5fwx-jx9f: Proof of Concept in advisory shows PyJWT accepts tokens with `crit: ["x-custom-policy"]` without rejection. Expected behavior: `InvalidTokenError: Unsupported critical extension`.
Upgrade to PyJWT v2.12.0 or later immediately. If upgrading is not possible, validate `crit` headers manually before decoding tokens.
2. Key confusion via PEM/SSH public keys
HIGH
DEPENDENCY_CVE ✓ Confirmed
PyJWT's HMAC algorithm does not block all PEM/SSH-encoded public keys, enabling attackers to use public keys as HMAC secrets. This allows forged tokens to be accepted when `algorithms.get_default_algorithms()` is used. Fixed in v2.4.0.
GHSA-ffqj-6fqr-9h24: PoC demonstrates HMAC validation with SSH-encoded Ed25519 public keys, bypassing signature checks. Affects versions 1.5.0–2.3.0.
Upgrade to PyJWT v2.4.0 or later. Always explicitly specify allowed algorithms (e.g., `algorithms=["HS256"]`) instead of using `get_default_algorithms()`.
3. PKCS1 PEM key confusion
HIGH
DEPENDENCY_CVE ✓ Confirmed
PyJWT versions ≤1.5.0 do not block PKCS1 PEM-encoded public keys in HMAC validation, allowing attackers to craft JWTs with symmetric keys derived from public keys. Fixed in v1.5.1.
GHSA-r9jw-mwhq-wp62: `invalid_strings` in `HMACAlgorithm.prepare_key` omits `-----BEGIN RSA PUBLIC KEY-----`, enabling key confusion attacks.
Upgrade to PyJWT v1.5.1 or later. Avoid using PKCS1 PEM keys for HMAC validation.
4. Issuer field partial matching bug
LOW
DEPENDENCY_CVE ✓ Confirmed
PyJWT v2.10.0 incorrectly uses string containment (`in`) instead of equality (`==`) for `iss` claim validation, allowing tokens with partial issuer matches to pass. Fixed in v2.10.1.
GHSA-75c5-xw7c-p5pm: PoC shows `issuer="urn:expected"` accepts `payload["iss"]="urn:"`. Regression introduced in commit 1570e708.
Upgrade to PyJWT v2.10.1 or later. Audit issuer validation logic in your codebase.
