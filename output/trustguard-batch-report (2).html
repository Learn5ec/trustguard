🔍 Security Findings 2 HIGH
1. CSRF vulnerability in FastAPI <0.65.2
HIGH
DEPENDENCY_CVE ✓ Confirmed
FastAPI versions below 0.65.2 are vulnerable to Cross-Site Request Forgery (CSRF) when using cookies for authentication in path operations receiving JSON payloads. The vulnerability arises from accepting JSON payloads with non-JSON content types (e.g., text/plain), bypassing CORS preflights and enabling CSRF attacks.
Vulnerability GHSA-8h2j-cgx8-6xv7: Affects versions 0.1.0 to 0.65.1, fixed in 0.65.2. References: FastAPI security advisory and NVD entry CVE-2021-32677.
Upgrade to FastAPI version 0.65.2 or later. If upgrading is not possible, implement a middleware or dependency to validate the Content-Type header strictly.
2. ReDoS vulnerability in form data parsing
HIGH
DEPENDENCY_CVE ✓ Confirmed
FastAPI versions below 0.109.1 are vulnerable to Regular Expression Denial of Service (ReDoS) when parsing form data. An attacker can craft a malicious Content-Type header that causes excessive CPU consumption, stalling the event loop and preventing further request handling.
Vulnerability PYSEC-2024-38: Affects versions 0.1.0 to 0.109.0, fixed in commit 9d34ad0ee8a0dfbbcce06f76c2d5d851085024fc (version 0.109.1).
Upgrade to FastAPI version 0.109.1 or later to mitigate the ReDoS vulnerability.

1. Critical SQL injection via group_by parameter
CRITICAL
DEPENDENCY_CVE ✓ Confirmed
SQLAlchemy versions up to and including 1.2.17 are vulnerable to SQL injection when the `group_by` parameter is user-controlled. This allows attackers to execute arbitrary SQL commands, leading to data breaches or unauthorized database access. The vulnerability is fixed in version 1.2.19.
Vulnerability ID GHSA-38fc-9xqv-7f7q; affected versions include all up to 1.2.17; fixed in 1.2.19.
Ensure the application uses SQLAlchemy version 2.0.30 or later, and avoid using user-controlled input in `group_by` parameters without strict validation or parameterized queries.
2. Critical SQL injection via order_by parameter
CRITICAL
DEPENDENCY_CVE ✓ Confirmed
SQLAlchemy versions before 1.3.0b3 are vulnerable to SQL injection via the `order_by` parameter. The fix was not backported to the 1.2.x line, leaving all 1.2.x versions vulnerable. This can allow attackers to manipulate SQL queries and access sensitive data.
Vulnerability ID GHSA-887w-45rq-vxgf; affected versions include all 1.2.x versions; fixed in 1.3.0b3.
Upgrade to SQLAlchemy version 1.3.0b3 or later. Avoid passing raw user input to `order_by` parameters; use allowlists or ORM query builders instead.
3. SQL injection via limit/offset parameters
CRITICAL
DEPENDENCY_CVE ✓ Confirmed
SQLAlchemy versions before 0.7.0b4 are vulnerable to SQL injection via the `limit` and `offset` parameters in the `select` function. This allows remote attackers to execute arbitrary SQL commands, posing a severe risk to data integrity and confidentiality.
Vulnerability ID GHSA-hfg2-wf6j-x53p; affected versions include all before 0.7.0b4; fixed in 0.7.0b4.
Ensure the application uses SQLAlchemy version 0.7.0b4 or later. Validate and sanitize all user input used in `limit` and `offset` parameters.

Secure Code Review

No source code was provided for direct review. However, the vulnerability data reveals critical implementation flaws in PyJWT's handling of JWT standards. Specifically, the library fails to validate the `crit` header parameter (RFC 7515 §4.1.11), allowing attackers to bypass security policies. Additionally, key confusion vulnerabilities (e.g., GHSA-ffqj-6fqr-9h24) enable attackers to craft malicious tokens using symmetric algorithms with asymmetric keys. 🔍 Security Findings 3 HIGH · 1 LOW
1. Unpatched crit header validation bypass
HIGH
DEPENDENCY_CVE ✓ Confirmed
PyJWT fails to validate the `crit` (Critical) Header Parameter as required by RFC 7515 §4.1.11. This allows attackers to include unrecognized critical extensions in JWTs, which the library accepts instead of rejecting. This can lead to security policy bypasses, token binding issues, and split-brain verification in mixed-library deployments.
Vulnerability GHSA-752w-5fwx-jx9f: Affects all versions <2.12.0. Proof of concept demonstrates acceptance of tokens with `crit: ["x-custom-policy"]` without rejection.
Upgrade to PyJWT v2.12.0 or later immediately. If unable to upgrade, explicitly validate the `crit` header in application code before decoding tokens.
2. Key confusion via non-blocklisted public keys
HIGH
DEPENDENCY_CVE ✓ Confirmed
PyJWT's HMAC algorithm does not block all PEM-encoded public key formats, enabling attackers to use asymmetric public keys (e.g., `ssh-ed25519`) as HMAC secrets. This allows attackers to craft valid JWTs from scratch, bypassing signature verification.
Vulnerability GHSA-ffqj-6fqr-9h24: Affects versions 1.5.0–2.3.0. Proof of concept demonstrates successful validation of HS256 tokens signed with an `ssh-ed25519` public key.
Upgrade to PyJWT v2.4.0 or later. Always explicitly specify allowed algorithms (e.g., `algorithms=["HS256"]`) instead of using `get_default_algorithms()`.
3. PKCS1 PEM key confusion in HMAC
HIGH
DEPENDENCY_CVE ✓ Confirmed
PyJWT's HMAC algorithm does not block PKCS1 PEM-encoded public keys (e.g., `-----BEGIN RSA PUBLIC KEY-----`), enabling symmetric/asymmetric key confusion attacks. Attackers can craft JWTs using publicly known keys to bypass signature checks.
Vulnerability GHSA-r9jw-mwhq-wp62: Affects versions ≤1.5.0. Fixed in v1.5.1, but similar logic flaws persist in later versions (e.g., GHSA-ffqj-6fqr-9h24).
Upgrade to the latest version and avoid using `get_default_algorithms()`. Enforce algorithm whitelisting in all JWT decoding operations.
4. Issuer field partial matching flaw
LOW
DEPENDENCY_CVE ✓ Confirmed
PyJWT v2.10.0 introduced a bug where the `iss` claim validation uses string containment (`in`) instead of exact matching (`==`). This allows tokens with partial issuer matches (e.g., `"urn:"` for `"urn:expected"`) to pass validation.
Vulnerability GHSA-75c5-xw7c-p5pm: Affects v2.10.0 only. Fixed in v2.10.1. Proof of concept demonstrates acceptance of `iss: "urn:"` when `issuer="urn:expected"`.
Upgrade to PyJWT v2.10.1 or later. Validate issuer claims manually if using v2.10.0.
