# Service Details for TrustGuard

## Overview
The **TrustGuard** project runs as a Vite development server on the local network and is exposed publicly via an Ngrok tunnel. Two systemd user services manage the lifecycle:

1. **trustguard‑app.service** – Starts the Vite dev server (`npm run dev`).
2. **trustguard‑ngrok.service** – Starts an Ngrok tunnel that forwards the local address `192.168.7.109:23232` to a public domain.

Both services are configured to start automatically after login and depend on each other (the Ngrok service waits for the app service).

## Systemd Unit Details
### trustguard‑app.service
```ini
[Unit]
Description=TrustGuard Vite Dev Server
After=network.target

[Service]
Type=simple
WorkingDirectory=/home/web-h-056/trustguard
ExecStart=/usr/bin/npm run dev
Restart=on-failure
RestartSec=5

[Install]
WantedBy=default.target
```
- **WorkingDirectory** points to the repository root.
- **ExecStart** runs the Vite dev server (`npm run dev`).
- The service is **enabled** and **started**: `systemctl --user enable --now trustguard-app.service`.

### trustguard‑ngrok.service
```ini
[Unit]
Description=Ngrok Tunnel for TrustGuard
After=network.target trustguard-app.service
Requires=trustguard-app.service

[Service]
Type=simple
ExecStart=/home/web-h-056/weblocker/node_modules/.bin/ngrok start --all --config /home/web-h-056/.config/ngrok/ngrok_trustguard.yml
Restart=on-failure
RestartSec=5

[Install]
WantedBy=default.target
```
- **ExecStart** points to the Ngrok binary installed under `weblocker/node_modules`.
- The service depends on `trustguard-app.service` so the Vite server is up before Ngrok opens the tunnel.
- The service is **enabled** and **started**: `systemctl --user enable --now trustguard-ngrok.service`.

## Ngrok Configuration (`ngrok_trustguard.yml`)
```yaml
version: "3"
agent:
  authtoken: <YOUR_NGROK_AUTHTOKEN>

# Tunnel definition
# This forwards the local Vite dev server (192.168.7.109:23232) to a public URL.
# The public URL is automatically generated by Ngrok and stored in the service logs.
# You can also set a fixed domain (as shown below).

tunnels:
  trustguard:
    proto: http
    addr: 192.168.7.109:23232
    domain: eastbound-unkempt-regulate.ngrok-free.dev  # optional fixed sub‑domain
```
- Replace `<YOUR_NGROK_AUTHTOKEN>` with the token you obtained from the Ngrok dashboard.
- The `domain` field is optional; if omitted Ngrok will assign a random sub‑domain.

## How to Use the Services
### Starting / Stopping
```bash
# Start both services (they are already enabled to start on login)
systemctl --user start trustguard-app.service
systemctl --user start trustguard-ngrok.service

# Stop them
systemctl --user stop trustguard-ngrok.service
systemctl --user stop trustguard-app.service
```
### Checking Status
```bash
systemctl --user status trustguard-app.service
systemctl --user status trustguard-ngrok.service
```
- The **app** service will report the Vite dev server listening on `http://192.168.7.109:23232`.
- The **ngrok** service will show the public URL (e.g., `https://eastbound-unkempt-regulate.ngrok-free.dev`).

### Logs
```bash
journalctl --user -u trustguard-app.service -f   # live logs for the Vite server
journalctl --user -u trustguard-ngrok.service -f # live logs for Ngrok
```

## Common Issues & Troubleshooting
| Symptom | Likely Cause | Fix |
|---------|--------------|-----|
| **Blank/black screen** when opening the public URL | Ngrok tunnel is not yet ready or the Vite server hasn't compiled assets. | Wait a few seconds, then refresh. Verify both services are `active (running)`. |
| **`ERR_NGROK_334`** – endpoint already online | Another Ngrok tunnel with the same domain is running (perhaps from a previous session). | Stop the existing Ngrok service: `systemctl --user stop trustguard-ngrok.service`. Then start it again. |
| **Port 23232 already in use** | Another process (maybe a previous Vite instance) is bound to the port. | Kill the stray process (`lsof -i :23232` → `kill <PID>`), then restart `trustguard-app.service`. |
| **Cannot connect to `192.168.7.109:23232` from another device** | Local firewall blocks inbound traffic. | Ensure the host firewall allows traffic on `23232` (e.g., `ufw allow 23232`). |
